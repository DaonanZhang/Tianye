第 1 章：项目概述与当前切口

对外展示正文

本项目定位为一款面向步行与徒步场景的助手产品，核心任务是帮助用户理解路线、完成路线，并将一次行走经历整理为可复用的路线内容。

项目当前不从“大而全的户外平台”切入，而是先围绕路线理解这一核心环节建立最小闭环，包括路线导入、轨迹解析、基础判断、路线说明与分享呈现。

早期验证样本将优先选择城市徒步路线，原因不是项目长期只做 Citywalk，而是城市场景更适合在早期获得相对密集、可整理、可复核的路线样本。这既有利于更快完成产品演示与用户反馈收集，也有利于在数据库和路线积累尚未形成规模之前，先建立一套可运行的样本基础。

概念边界

`户外徒步助手` 是产品定位；`轻户外` 是更大的需求赛道；`城市徒步` 是当前验证场景；`Citywalk` 更接近一种内容表达和传播语境。四者相关，但不能混用。

本计划书将以“产品定位”作为主线，以“城市徒步切口”解释早期路径，以“Citywalk 语境”说明为什么这个切口更容易形成初期差异化。

Cautious Panel / 谨慎判断

如果后续访谈发现用户对“Citywalk”一词的理解过于偏内容消费、而非路线使用，正式外发版本可以进一步降低这个词在正文中的占比，只保留“城市徒步”与“轻户外”两个层级。

一句话介绍 一个帮助用户理解、完成和分享徒步路线的助手产品。
阶段定义 Demo / MVP 准备期，先证明路线理解闭环在城市场景中可运行。

第 2 章：用户场景与问题定义

对外展示正文

目标用户并不缺少路线信息，真正缺少的是一套能够围绕“出发前判断、途中参考、走后沉淀”这三个环节连续工作的产品体验。当前常见情况是：用户先在内容平台上看到一条被推荐的路线，对路线产生兴趣；随后切换到地图工具补充交通、入口和周边信息；如果还想进一步确认轨迹，又需要去户外工具里寻找 GPX 或现成路线记录。信息并非不存在，问题在于它们分散在不同类型的平台里，彼此之间缺少衔接。

这种分散会直接带来决策断裂。内容平台擅长激发兴趣，但往往缺少地图化规划、轨迹结构和可复走能力；地图工具擅长导航和地点搜索，但很少把城市徒步或轻户外路线作为核心内容组织对象；专业户外工具能够提供轨迹、里程和记录能力，但对普通用户而言仍然偏重，尤其在城市轻徒步场景下，路线理解、路线挑选和内容表达并不够轻量。用户最终看到的是很多信息，却仍然缺少一个专门为 Citywalk 或轻户外徒步决策服务的统一工具。

对普通用户而言，路线决策最关键的问题通常集中在四点：是否好走、是否值得走、是否适合当前体能与时间安排、以及走完以后如何保存或分享。现有信息供给能够提供灵感，也能提供局部能力，但并不稳定地把这四个判断组织成一条顺畅路径。

因此，本项目要解决的不是“路线从零到一不存在”的问题，而是“路线存在，但理解成本和复用成本过高”的问题。

为什么要区分这三个概念

区分 `轻户外`、`城市徒步` 与 `Citywalk`，不是为了制造新概念，而是为了避免产品定义失焦。对本项目而言，`轻户外` 负责定义赛道边界，`城市徒步` 负责定义当前场景，`Citywalk` 只在涉及内容表达和传播时使用。

Cautious Panel / 平台举例与证据写法

这一章可以在正式版本中出现平台名称，但建议作为“用户现有行为路径”的举例，而不是把平台写成攻击对象。更稳妥的写法是使用“内容平台、地图工具、专业户外工具”等类型词为主，必要时再辅以 1 到 2 个具体平台名。后续如果补证据，优先补用户跨平台完成一次路线决策的真实流程截图或访谈原话，而不是只补热度趋势。

[INTERNAL] 这里可以补的证据类型： - 城市徒步 / Citywalk 的近年内容热度趋势 - 周末轻户外、周边游、城市漫步的消费场景变化 - 用户为什么在多个平台来回切换 - 这些词在用户语境中的差别：谁更偏产品本质，谁更偏赛道，谁更偏路线，谁更偏内容传播 后续如果要更强版本，可以补 3 类真实材料： 1. 典型小红书路线笔记截图 2. 公众号/文旅路线页截图 3. 专业 GPX 或户外工具截图 你的重点不是证明“大家爱徒步”这么宽泛，而是证明： “发现路线”和“真正复走路线”之间存在明显产品断层。

第 3 章：用户痛点拆解

对外展示正文

现在这类用户最核心的问题不是“没有路线”，而是“路线信息很难快速变成行动”。问题主要集中在四步：选路线、看懂路线、跟着走、走完以后怎么整理和分享。对早期项目来说，这也是好事，因为问题足够具体，可以先做一个小而清楚的切口来验证，而不是一开始就做成大而全的平台。

痛点 1 新手不知道从哪条路线开始，无法快速判断路线是否适合自己。
痛点 2 图文攻略可以种草，但很难直接变成“今天就能跟着走”的路线工具。
痛点 3 GPX 对专业用户有价值，但对普通用户的理解和使用门槛偏高。
痛点 4 现有专业户外工具更强调功能完整度，不一定适合城市轻徒步新手。

此外，用户在走完一条路线后也面临“二次表达成本”过高的问题。很多人会留下照片、轨迹和片段化感受，但缺少一个低成本方式把这些信息整理成可复用、可分享、可被他人继续参考的路线卡片或路线故事。

这意味着产品价值不仅体现在“帮用户找路”，也体现在“帮用户把一次徒步沉淀成新内容”，从而形成路线供给的正循环。

[INTERNAL] 这一章可以准备“真实用户语言”版本，后面面谈会更有说服力。 建议你自己先写 8-12 句用户原话模板，例如： - “看了攻略还是不知道入口在哪” - “下载 GPX 之后不知道怎么打开” - “地图能搜地点，但不能告诉我这条路线值不值得走” - “我走完了，但懒得整理成一篇完整笔记” 之后这章可以升级成“用户痛点 -> 产品动作”映射表： - 不知道选哪条 -> 提供轻量筛选与路线卡片 - 看不懂轨迹 -> 提供结构化路线说明 - GPX 有门槛 -> 提供上传即解析 - 分享成本高 -> 提供自动生成文案和卡片

第 4 章：市场缺口与竞品观察

对外展示正文

路线发现、路线记录和路线分享并不是新需求，市场上已经存在内容平台、地图平台、专业户外工具和国际路线社区等多种成熟形态。对早期项目而言，重点不在于证明“市场是否存在”，而在于识别哪些环节仍然存在明显缺口。

当前最值得关注的缺口，是普通用户从“看到路线”到“决定去走”之间仍然缺少一层足够清晰、足够轻量的路线理解工具。内容平台擅长激发兴趣，专业工具擅长提供能力，但两者之间仍有产品空位。

竞品类型 优势 不足 本项目切入点
小红书 / 公众号 / 文旅内容 内容生动、传播强、容易种草 路线不结构化、难复走、难持续维护 把图文灵感转化为可跟走路线
高德 / 百度地图 搜索、导航、交通信息完整 路线故事与社区表达不是核心 补足路线内容化与场景解释层
两步路等户外工具 轨迹、活动、社区、专业功能较强 新手门槛偏高，城市轻徒步表达不够轻 面向普通用户重新包装路线理解体验
AllTrails / Komoot 路线发现与社区产品成熟 国内本地化和中文文旅语境适配有限 更贴近中文城市漫步和本地路线生态

本项目与两步路等专业户外工具并非完全错位。长期来看，项目同样希望服务更完整的徒步场景；但在第一阶段，不会直接与成熟产品竞争功能完整度，而是优先建立“路线理解更轻、新手门槛更低、内容表达更清楚”的早期差异化。

从执行角度看，城市徒步切口的意义也很明确：一方面，城市路线的公开信息、轨迹来源、地点信息和内容素材相对更容易搜集与校验，能够缓解项目早期在路线数据库积累上的天然劣势；另一方面，这类样本也更容易转化为可展示 Demo，更适合单人创始阶段完成验证。

Cautious Panel / 竞争判断

如果后续发现用户真正强需求并不在“城市路线理解”，而更接近“近郊徒步安全判断”或“路线筛选效率”，竞品对比矩阵需要同步调整，不能把“城市”误写成永久主战场。

第 5 章：产品方案与核心价值

对外展示正文

项目的基本解法，是将原本偏技术化的路线数据转化为普通用户可以直接理解和使用的路线辅助结果。用户既可以查看现成路线，也可以导入 GPX 文件；系统负责完成轨迹解析、基础判断、路线说明与分享呈现。反过来，用户在完成一次路线体验之后，也可以将自己的路径整理并导出为 GPX 文件，继续用于保存、分享或二次复走。

从产品结构看，核心闭环并不复杂：用户可以从现成路线进入，也可以从自有轨迹进入；在使用过程中完成路线理解与判断；在结束之后再把这次路径沉淀为可继续流通的路线内容或 GPX 文件。早期阶段的目标，就是把这条输入、理解、使用、再输出的链路做短、做清楚、做成可验证的产品动作。

核心价值主张

  • 提供更易理解的路线辅助能力，让非专业用户也能快速判断一条路线是否适合自己。
  • 降低路线复走门槛，让图文灵感和轨迹文件都能更直接变成可执行路线。
  • 降低路线分享门槛，让用户完成一次徒步后更容易产出结构化内容。
  • 为后续社区化、推荐化和商业化积累可复用的路线数据资产。

第一阶段产品流程

  1. 用户选择一条测试路线，或上传本地 GPX 文件。
  2. 系统解析轨迹并在地图中展示路线。
  3. 系统输出距离、时长、闭环、难度等基础信息。
  4. 系统生成路线说明、亮点、适合人群与注意事项。
  5. 系统生成可复制、可展示、可分享的路线卡片。
[INTERNAL] 可以把本章理解为“产品闭环图”的文字版。 后续如果你做 PPT，这一章最适合画成 5 个节点： 路线输入 -> 轨迹解析 -> 路线分析 -> 内容生成 -> 分享沉淀 建议在口播里强调： - 我们不是让用户多学一个专业工具 - 我们是在消化专业格式，再还给用户一个轻量结果 本项目长期可能存在的三层价值： 1. 徒步助手 / 路线辅助层 2. 路线内容层 3. 路线数据资产层 这个三层逻辑很重要，后面商业化和技术壁垒都可以从这里往外长。 但对外时要记住： - 现在先讲清楚第 1 层和第 2 层已经如何开始验证 - 第 3 层是方向，不要包装成已经形成的数据壁垒

第 6 章：首次 Demo 与 MVP 规划

对外展示正文

项目第一阶段优先完成一个 Web Demo，用最短路径证明核心产品闭环，而不是在早期就追求完整 App。这个 Demo 的目标很明确：在 3 到 5 分钟内，让观察者看懂产品价值并看到实际可运行结果。

首次 Demo 阶段不以商业收入为验证目标。现阶段优先验证用户是否愿意使用这套产品完成路线理解、复走与分享；商业化路径将在核心使用行为得到初步验证后再逐步展开。

首次 Demo 的核心功能

  • 路线列表页：展示 5 到 10 条北京城市徒步测试路线。
  • 路线详情页：展示地图轨迹、基础数据、路线说明和卡片区域。
  • GPX 上传功能：支持用户导入本地 GPX 并立即查看分析结果。
  • 路线分析功能：计算距离、闭环状态、预计用时和简单难度。
  • 路线说明生成功能：使用规则模板或 mock AI 先完成内容层闭环。
  • 路线卡片功能：把路线数据与路线文案整理成可分享呈现。

Demo 阶段明确不做

首次 Demo 不做登录注册、不做复杂社区、不做支付、不做实时 GPS 记录、不做原生手机 App、不做完整推荐系统，也不依赖真实商业化收入逻辑。目标只有一个：先让核心闭环成立。

第 7 章：智能化能力与技术路径

对外展示正文

本项目希望做的，不只是一个静态路线工具，而是一个带有智能辅助能力的徒步助手。对普通用户而言，真正有价值的并不是单纯看到轨迹，而是获得一份更接近“有人提前帮你读过这条路线”的判断结果：这条路线适不适合当前体能，亮点在哪里，可能会遇到什么问题，是否值得在当前时间出发。

因此，AI 与智能化能力在本项目中的作用，不是额外附着的一层包装，而是把原始路线数据转化为可理解、可决策、可分享内容的关键环节。项目希望逐步形成的，不只是轨迹展示能力，而是路线解释、路线摘要、适合人群判断、注意事项生成，以及后续的个性化推荐能力。

从这个角度看，项目的智能化能力可以理解为三层：第一层是把 GPX、距离、海拔、轨迹点等原始数据转化为结构化结果；第二层是把结构化结果转化为普通用户能直接理解的自然语言说明和路线卡片；第三层是在积累更多路线与用户反馈之后，进一步形成更贴近个人需求的推荐与路线判断能力。

第一阶段的重点仍然是把路线数据稳定地转化为结构化说明，并完成基础的路线内容生成；在这一步成立之后，再逐步强化模型生成、风格适配和推荐能力。这样写的原因很简单：项目的 AI 价值必须建立在真实路线数据、真实使用场景和稳定解释流程之上，而不是停留在“会写一段文案”的层面。

第 8 章：目标用户与典型使用场景

对外展示正文

项目第一阶段主要面向四类用户：城市轻徒步新手、路线分享者、偏内容表达的 Citywalk 用户，以及对轨迹真实性和路线结构有更高要求的进阶用户。前两类是当前最核心的验证对象；后两类帮助项目观察内容传播和高质量供给的可能性。

用户概念分层

在用户分层上，`轻户外` 是外层赛道概念；`城市徒步用户` 是当前最接近产品核心需求的人群；`Citywalk 用户` 则更偏向内容表达、传播与城市体验。一个用户可能同时属于这三类，但产品当前优先服务的是“需要路线判断”的人，而不是只需要内容灵感的人。

核心用户画像

  • 城市轻徒步新手：希望周末走 3 到 8 公里，重视轻松、安全、交通便利与清晰说明，更关心这条路线能不能顺利完成。
  • Citywalk 用户：关注路线氛围、街区体验、拍照点与故事感，希望路线可以保存和分享，更关心这条路线有没有表达和传播价值。
  • 路线分享者：愿意上传轨迹、照片与经验，但不愿意花大量时间整理完整攻略。
  • 进阶用户：关注轨迹真实性、海拔、补给点、风险点与复盘能力，可成为高质量内容贡献者。

典型使用场景

一个典型场景是：用户在周末前寻找一条 3 到 8 公里、交通方便、强度可控的北京路线。用户进入产品后，快速浏览测试路线，查看轨迹、距离、预计时间、亮点和注意事项；如果路线合适，可以直接保存或转发给同行者。走完之后，用户还可以上传自己的 GPX，生成新的路线卡片。

项目当前并不以正式融资为重点。现阶段的关键任务，是把产品验证、展示材料、路线样本和阶段性资源需求整理清楚，为后续产品推进、合作沟通和外部支持对接打下基础。

Cautious Panel / 对外口径

这一章应以“资源需求与合作支持”为主，而不是以融资叙事为主。对外表达时，可以优先强调样本共建、场景合作、产品验证支持和基础创业配套，而不是预设项目已经进入正式融资流程。

当前优先需要的资源

  • 产品验证资源：目标用户访谈、试用反馈和路线内容样本补充。
  • 创业配套资源：公司注册、注册地址、财税与合规咨询。
  • 展示与沟通资源：商业计划书、演示材料和路演表达优化。
  • 场景资源：与文旅、体育、城市服务、路线活动相关的潜在合作对接。
  • 后续组织资源：当验证成立后，支持寻找设计、内容或移动端协作补位。

如果获得外部支持，优先投入方向

  • Demo 向 MVP 的产品开发与设计补强。
  • 测试路线整理、内容样本建设与用户验证执行。
  • 基础公司运营与必要的合规成本。
  • 早期小规模增长实验，而非大规模买量。

第 14 章：风险、边界与下一步行动

对外展示正文

作为一个仍处于早期验证阶段的项目，本项目对当前边界有明确判断：第一目标不是同时完成完整社区、完整导航、完整商业化或原生移动端，而是通过一个简洁、可信、可运行的产品闭环证明路线数据与路线内容结合的价值。

项目当前面临的主要挑战包括：初始路线内容冷启动、早期用户访谈组织、展示材料与产品体验的一致性，以及从 Web Demo 过渡到移动端形态时的架构演进。

在此基础上，下一步动作也很清楚：完成最小 Demo，整理路线样本与展示材料，开展小范围试用，并根据反馈调整 MVP 范围。

下一步行动清单

  1. 完成 Demo 必要功能并形成可稳定演示版本。
  2. 补齐北京测试路线、截图、演示脚本与展示材料。
  3. 完成首轮目标用户访谈与试用记录。
  4. 根据反馈压缩或增强 MVP 功能范围。
  5. 按真实需要推进公司注册、园区沟通与政策准备。
[INTERNAL] 这章本质上是“收口”。 建议你以后对外时一定要保持一个很清楚的边界感： - 我们知道自己现在不是完整产品 - 但我们也不是纯概念 - 我们正在用一个可展示闭环来验证最核心的价值主张 这比一上来把未来讲得太满更可信。 你后续还可以在这一章补一个“风险应对表”： - 冷启动内容不足 -> 先由创始人和公开路线样例填充 - 用户增长慢 -> 先做深小样本反馈 - 真实 AI 接口成本高 -> 先规则模板，保留替换接口 - 移动端迁移成本 -> 当前模块设计保持可移植性
