# 北京城市徒步路线 AI Demo 开发说明

## 0. Demo 目标

本 Demo 的目标不是做完整 App，而是做出一个可以首次展示的核心产品闭环：

> 选择或上传一条 GPX 徒步路线 -> 在地图上展示轨迹 -> 自动分析路线基础信息 -> 生成 AI 风格路线介绍 -> 生成一张可分享的路线卡片。

这个 Demo 用来证明三件事：

1. 项目不是纯 PPT，而是已经有可运行的产品原型。
2. GPX 徒步轨迹可以被解析、可视化和内容化。
3. AI 可以把冷冰冰的路线数据转化成普通用户愿意阅读和分享的路线内容。

当前阶段不要追求完整产品，不要做复杂社区，不要做真实商业化系统。

## 1. Demo 的产品定位

Demo 名称暂定为：

> 北京城市徒步路线 AI Demo

一句话介绍：

> 一个可以把 GPX 徒步轨迹自动转换成路线介绍、路线亮点和分享卡片的 AI 徒步路线社区原型。

目标用户：

- 城市徒步用户
- 周末轻户外用户
- 想找 3-10 公里城市路线的人
- 想记录并分享自己路线的人
- 对 GPX 有兴趣，但不想处理复杂技术细节的普通用户

首次展示时要让观看者理解：

> 这不是一个普通地图工具，而是一个把路线数据变成社区内容的产品。

## 2. 技术实现默认假设

优先做 Web Demo，不做原生 App。

默认技术方向：

- 前端：React / Vite 或当前项目已有前端框架
- 地图：Leaflet + OpenStreetMap
- 数据：本地 GPX 文件 + 本地 JSON 配置
- 后端：第一版可以没有后端
- AI：第一版先做可替换的 mock AI / rule-based generator，不强依赖真实 LLM API

如果当前仓库已经有固定技术栈，请优先遵守现有项目结构，但功能边界保持不变。

## 3. 最小 Demo 必须完成的核心流程

### 3.1 路线列表页

路径建议：`/`

页面目标：让用户第一眼知道这是一个“路线社区”。

必须展示：

- Demo 标题
- 一句话产品介绍
- 6 条测试路线卡片
- 每张卡片展示路线名称、城市、简短描述、标签
- 点击卡片进入路线详情页

不需要实现：

- 登录
- 搜索
- 筛选
- 收藏
- 评论
- 真实社区动态流

验收标准：

- 打开首页后可以看到所有 demo route
- 点击任意 route 可以进入详情页
- 首页视觉上要像一个产品入口，而不是开发调试页面

### 3.2 路线详情页

路径建议：`/routes/:routeId`

页面目标：展示一条路线的完整体验。

必须展示：

- 路线标题
- 地图轨迹
- 路线基础信息
- AI 路线介绍区域
- 生成路线卡片按钮

路线基础信息至少包括：

- 总距离
- 预计用时或实际用时
- 起点坐标
- 终点坐标
- 是否闭环路线
- 停留点数量
- 简单难度：轻松 / 适中 / 较难

验收标准：

- 详情页能加载对应 GPX
- 地图上能画出完整轨迹
- 能展示基础统计信息
- 页面不因为某条 GPX 缺少时间字段而崩溃

### 3.3 GPX 解析模块

模块目标：把 GPX 文件转换成前端可以使用的路线数据。

需要支持：

- 读取 GPX XML
- 解析 `trkpt` 节点
- 提取 latitude / longitude
- 尽量提取 elevation
- 尽量提取 timestamp
- 对缺少 elevation 或 timestamp 的 GPX 做兼容

需要避免：

- 解析失败后整个页面白屏
- 把解析逻辑直接写在组件里
- 大量重复代码

验收标准：

- 每条本地 GPX 都可以被解析
- 解析失败时展示清楚的错误信息
- 解析逻辑被拆成独立 helper function

### 3.4 路线分析模块

模块目标：从 GPX 点位中计算最基本的路线信息。

第一版必须实现：

- 使用 Haversine 公式计算总距离
- 根据起点和终点距离判断是否闭环
- 根据总距离给出简单难度
- 尝试识别停留点
- 如果时间数据不足，则用估算用时

难度规则可以先简单处理：

- 小于 5km：easy
- 5km 到 10km：medium
- 大于 10km：hard

闭环判断可以先简单处理：

- 起点和终点距离小于 300 米，认为是闭环路线

停留点判断可以先简单处理：

- 如果 GPX 有时间字段：同一区域内长时间移动距离很小，认为可能是停留点
- 如果 GPX 没有时间字段：可以暂时返回空数组
- 不要为了停留点识别引入复杂算法

验收标准：

- 距离计算结果大致合理
- 闭环判断能正常显示
- 缺少时间字段时系统不崩溃
- 所有分析逻辑都独立于 UI 组件

### 3.5 AI 路线介绍模块

模块目标：把路线分析结果转化成普通用户能理解的内容。

第一版不强制接入真实 LLM。先实现一个可替换的 mock generator。

需要输出：

- 路线标题
- 一句话简介
- 适合人群
- 路线亮点
- 难度描述
- 注意事项
- 分享文案

生成逻辑第一版可以基于：

- route name
- distance
- difficulty
- loop route
- stop point count
- tags

验收标准：

- 点击“生成 AI 路线介绍”后可以展示完整文案
- 不需要真实 API key
- 后续可以替换成 OpenAI / 其他 LLM provider
- AI 生成逻辑不要写死在页面组件里

### 3.6 路线卡片生成模块

模块目标：把路线分析和 AI 文案合并成一个可展示、可截图、可分享的路线卡片。

路线卡片必须展示：

- 路线名称
- 城市
- 总距离
- 预计用时
- 难度
- 路线亮点
- 一句话简介
- 小地图或路线缩略图
- “AI 生成”标识

第一版不需要真的生成图片文件。只需要页面上有一个视觉上完整的 card section。

可以提供按钮：

- 生成路线卡片
- 复制分享文案

验收标准：

- 用户可以从详情页生成路线卡片
- 路线卡片视觉上适合用于展示
- 分享文案可以复制到剪贴板
- 复制失败时有友好提示

### 3.7 GPX 上传页

路径建议：`/upload`

页面目标：证明产品可以处理用户自己的路线，而不是只能展示写死数据。

第一版功能：

- 用户选择本地 `.gpx` 文件
- 前端读取文件内容
- 解析 GPX
- 地图展示轨迹
- 展示路线基础分析
- 生成 AI 路线介绍

限制：

- 不需要保存上传结果
- 不需要用户登录
- 不需要上传到服务器
- 不需要支持多个文件同时上传

验收标准：

- 用户可以上传一个 GPX 文件并看到路线
- 上传非 GPX 文件时有错误提示
- 空文件或损坏文件不会导致页面崩溃

## 6. 展示脚本

首次展示时按这个流程演示：

1. 打开首页，展示“北京城市徒步路线 AI Demo”。
2. 说明这里有几条北京城区徒步路线。
3. 点击“奥森公园环线”。
4. 展示地图轨迹和基础路线数据。
5. 点击“生成 AI 路线介绍”。
6. 展示 AI 生成的标题、亮点、适合人群和注意事项。
7. 点击“生成路线卡片”。
8. 展示可分享路线卡片。
9. 打开 GPX 上传页。
10. 上传一条测试 GPX，证明系统可以处理外部路线文件。

整个展示控制在 3-5 分钟。

## 7. 代码质量要求

开发时请遵守以下要求：

1. 所有核心逻辑都应该拆成 helper function，不要全部写在 UI 组件里。
2. GPX 解析、路线分析、AI summary generator 必须是独立模块。
3. 变量命名必须清楚，不要使用 `cb`、`el` 这类缩写。
4. JavaScript / TypeScript 中避免使用全局变量。
5. 尽量避免 inline style，优先使用 CSS class。
6. 代码注释使用 English。
7. 如果使用 Python，必须有 type hints 和 docstring。
8. 如果使用 TypeScript，核心数据结构需要定义 `interface` 或 `type`。
9. 错误状态必须有用户可理解的提示。
10. 完成功能后，询问是否需要为核心 helper functions 添加测试。

## 8. 最终验收标准

这个最小 Demo 完成后，应该能满足下面的验收标准：

1. 首页能展示 6 条测试路线。
2. 点击任意路线可以进入详情页。
3. 详情页能在地图上显示 GPX 轨迹。
4. 系统能显示总距离、起点、终点、难度等基础信息。
5. 用户可以点击按钮生成 AI 风格路线介绍。
6. 用户可以生成一张视觉完整的路线卡片。
7. 用户可以上传本地 GPX 文件并查看路线分析结果。
8. 缺少时间字段、海拔字段或 GPX 异常时，页面不会崩溃。
9. 代码结构清晰，核心逻辑不直接堆在组件里。
10. 整个 Demo 可以在 3-5 分钟内完整展示。

## 9. 最重要的边界提醒

请不要把这个 Demo 做成完整产品。

当前 Demo 只需要证明一个核心闭环：

> GPX 路线数据可以被解析、展示、分析，并通过 AI 风格文案变成可分享的徒步路线内容。

只要这个闭环顺畅，就已经达到了首次展示目的。
